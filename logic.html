How feed → csv works.

A two-page walkthrough of the streaming XML-to-CSV pipeline — what happens between paste-the-URL and download-the-file, and why each piece exists. Useful when a feed misbehaves or when someone asks whether the tool can handle a 1 GB upload.

1What it is

A static webpage backed by a single Cloudflare Worker. Drop in a feed URL, an uploaded .xml file, or pasted XML, and out comes a CSV — one row per record, one column per field. No login, no account, nothing stored. The whole thing fits in one HTML file plus a ~100-line worker.

2Architecture

Two components — a server-side proxy (≈100 lines, runs on Cloudflare's edge) and an in-browser streaming parser (≈500 lines of vanilla JS in index.html). The browser does the heavy lifting; the worker only exists to dodge CORS.

┌─────────────┐ https ┌──────────────────────┐ https ┌────────────────────┐ │ Browser │──────────────▶│ Cloudflare Worker │──────────────▶│ Upstream feed │ │ index.html │ /api/feed │ _worker.js (~100ln) │ passthrough │ S3 / CDN / origin │ │ │◀──────────────│ streams body back │◀──────────────│ XML, often >1 GB │ └─────────────┘ chunked └──────────────────────┘ chunked └────────────────────┘ │ ▼ parse in browser ┌─────────────────────────────────────────────┐ │ Streaming XML parser → CSV chunks → Blob │ │ detectRecordTag() parseRecord() emitRow()│ └─────────────────────────────────────────────┘ │ ▼ feed.csv (download)

Why the worker exists at all

Pure CORS. Most job-feed URLs (S3, CDNs, customer origins) don't return Access-Control-Allow-Origin: *, so the browser refuses to read them. The worker fetches server-side and re-emits the response with the right header. It never buffers — the upstream body streams straight through. Cloudflare's free tier has no wall-clock cap on streaming I/O, which is why 1+ GB feeds work.

Why parsing is in the browser

  • Privacy — no feed contents ever sit on a server we own.
  • Cost — Cloudflare Worker CPU time is metered; browser CPU is free.
  • Simplicity — one static page, no backend state.

3The pipeline (paste URL → CSV)

Step 1 — Fetch with fallback chain

The URL tab tries three fetch strategies in order, taking whichever one succeeds first:

  1. /api/feed?url=<target> — our Cloudflare worker proxy
  2. api.allorigins.win — public CORS proxy fallback
  3. direct fetch(target) — only works if the upstream sets CORS

Step 2 — Detect the record tag

The parser doesn't assume the records are called <job>. It looks at the first ~64KB and counts every tag. The most-repeated tag wins. Before counting, it strips four things that would otherwise pollute the count:

StrippedWhy
<?xml … ?>processing instruction (always exactly 1)
<!-- … -->comments (could contain anything)
<!DOCTYPE …>doctype declarations
<![CDATA[ … ]]>HTML inside descriptions — was breaking real feeds
Real bug fixed 2026-05-13: A Joveo feed had 14 <job> records and 476 <br> tags inside <description> CDATA. Pre-fix, the detector picked <br> as the "most common tag" and produced zero rows. Stripping CDATA before counting fixed it.

Step 3 — Stream-parse records

Once the record tag is known, the parser scans the incoming buffer for open/close pairs. Each matched block is handed to DOMParser which extracts:

  • All attributes on the record (prefixed with @)
  • All direct-child elements' text content (whitespace-collapsed)
  • Repeated children joined with ;

Records get popped off the buffer as soon as they're parsed, so memory never grows beyond ~one record's worth of XML.

Step 4 — Lock schema after 50 rows

Feeds aren't always consistent — record 1 might have 12 fields, record 32 might add a 13th. The parser holds the first 50 rows in memory, accumulates the union of all field names, then writes the CSV header and flushes those buffered rows. From row 51 onward, schema is locked and emit-as-you-go.

Step 5 — Flush CSV chunks to Blobs

Naïve "build the CSV in a string" hits V8's ~512 MB string limit around 600K rows. The fix: every ~1 MB of CSV text gets flushed into a Blob and pushed to an array. At the end: 

new Blob([chunk1, chunk2, …, chunkN])

Blobs are backed by file-system-friendly storage, so the final download object can be 1 GB+ without ever holding it all in JS string memory.

5Cache + deploy gotcha

Cloudflare's static-asset binding aggressively caches HTML — by default the browser can hold an old index.html for an hour after a fresh deploy. The worker forces no-store on HTML responses so a deploy lands on the very next request: 

cache-control: no-store, no-cache, must-revalidate, max-age=0
cf-cache-status: BYPASS

Non-HTML assets (favicon, etc.) still cache normally. This was hot-fixed after the CDATA-detection rollout because operators were seeing the old parser despite a successful deploy.

6Limits

  • Free-tier requests — 100,000 worker invocations per day. Each conversion is one invocation.
  • Concurrent users — no soft limit; each browser does its own parse.
  • Browser memory — the Blob-chunking pattern was tested to ~1.5 GB CSV output on a 16 GB laptop. Above that, the browser may swap.
  • Wall-clock — proxy stream has no cap (Cloudflare property), so a slow upstream just means a slow parse. The UI shows live byte/row counters.
  • Non-XML inputs — surfaced as a parse error per record; the schema-discovery phase will time out gracefully with an empty result rather than crash.

7Data handling

  • The Cloudflare Worker proxies bytes; it does not buffer or log feed contents.
  • All XML→CSV parsing runs in your browser tab.
  • The download is generated as a local Blob and never round-trips back to a server.
  • No analytics, cookies, or third-party scripts.

9FAQ

Why doesn't the URL tab work for some feeds?

The upstream is either rejecting the worker's user-agent, requiring authentication, or returning a non-XML body. The status bar will say "upstream returned 403" or similar. Try the File tab — download the feed locally first, then drag it in.

What if the record tag is something weird?

The tag-stat box on the result panel shows what the parser picked. If it's wrong, the most common cause is the feed having a one-off structure where the "wrong" tag genuinely repeats more than the right one. Open a GitHub issue with a sample.

Why CSV instead of JSON or Parquet?

Excel and Google Sheets open CSV directly. Most operator workflows end at "drop it in a sheet" so CSV is the lowest-friction format.

Can it convert CSV back to XML?

Not today. The use case has been one-way (raw feed → tidy table) so the reverse hasn't come up. Adding it is straightforward — file an issue if you need it.
